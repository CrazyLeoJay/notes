# SDK 框架搭建

根据引言中的内容，可以构建以下的框架。

![手游SDK模块.drawio](framework.assets/%E6%89%8B%E6%B8%B8SDK%E6%A8%A1%E5%9D%97.drawio.svg)

## 整体设计

首先，根据业务可以了解到，手游的开发场景，存在同一sdk服务对应多个对外接口，例如发行APi、聚合API等，有些时候，手游API也会直接对外开放，不一定拘泥于这三种名称，这一行，统称为**对外APi开放层**。

然后，第二层可以称为**实现层**，涉及到具体实现，在具体实现里，也分为两大部分，类似于树状，分为主干的SDK Core和其他细分模块，可以理解为枝叶。也有一些模块设计为可拓展的，方便后续多人开发时优化接入流程，可以理解为较为粗壮的分叉。

当然，这个架构需要根据实际情况，如果业务没有那么复杂，可以直接将对外手游API于SDK Core 写于一起，当作一个上游api模块。



## 对外API开放层

这一层是对手游SDK的一层封装，对外接口也和手游SDK的api依次对应，不同的是，会在调用APi时，采集调用的数据，对相应的服务器进行数据上报，或者说记录。



## 手游SDK

这一层封装了具体实现的功能，包括但不限于：登录、注册、支付、悬浮球、数据上报、手游活动等游戏接入功能。也包含一些插件接口模块，比如：三方媒体数据上报模块、三方应用市场接入模块、三方其他可配置SDK接入等。

[具体的设计方案参考文档](design)

## 三方SDK实现

其他可以配置SDK，根据设计好的接口模块接入后，可以做成独立的项目，根据实际业务需求进行添加或者减少。
